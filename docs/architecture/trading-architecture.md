# Trading Architecture

Status: Draft
Date: 2026-03-07
Scope: 프로젝트 신규 참여자가 트레이딩 시스템의 핵심 제약, 처리 시나리오, 확장 경계를 빠르게 이해하기 위한 상위 설계 문서

## 1) 이 문서의 성격

이 문서는 구현 상세를 설명하는 코드 해설서가 아니라, 시스템이 어떤 책임 단위로 나뉘어야 하는지와 어떤 제약 아래 동작해야 하는지를 설명하는 설계 문서다.

따라서 여기서 말하는 컴포넌트는 특정 클래스나 파일 이름을 뜻하지 않는다.
구현은 바뀔 수 있지만, 아래 원칙과 경계는 쉽게 바뀌지 않는 것을 목표로 한다.

## 2) 이 시스템이 하는 일

이 프로젝트의 트레이딩 시스템은 다음을 반복 수행한다.

1. 시장/종목 데이터를 읽는다.
2. 전략 규칙으로 매수/매도 여부를 판단한다.
3. 주문을 브로커 API로 전송한다.
4. 주문/잔고/가격 상태를 다음 루프를 위해 갱신한다.

운영자는 `order_ui`로 종목별 주문 파라미터를 조정할 수 있고, 시스템은 정해진 동기화 시점에 이를 반영한다.

## 3) 이 프로젝트의 설계 제약

이 시스템은 범용 트레이딩 플랫폼이 아니라, 다음 제약을 전제로 설계한다.

### 3.1 외부 환경 제약

- 브로커는 복수일 수 있으며, 브로커마다 API 형태와 시장 규칙이 다를 수 있다.
- 외부 브로커는 서버 장애, 응답 지연, 일시적 비가용 상태를 가진 의존성으로 취급한다.
- 브로커의 REST API 호출 허용량이나 세션 정책은 명시적으로 공개되지 않을 수 있다.

### 3.2 내부 운영 원칙

- 상위 계층은 브로커별 세부 API 차이를 직접 알지 않는다.
- 시스템은 명시되지 않은 외부 제약을 고려해 보수적 호출과 경험적 운영 기준을 전제로 동작한다.
- 전략은 교체 가능해야 하지만, 실행 엔진 전체를 다시 작성하지 않고 교체할 수 있어야 한다.
- 운영 입력은 사람이 수정할 수 있는 외부 채널을 통해 들어오며, 실시간 즉시 반영보다 안전한 시점 반영을 우선한다. (현재는 엑셀 기반)
- 기본 운영은 `crontab` 등 외부 스케줄러를 통한 일 단위 실행을 전제로 하며, 별도 장치가 없으면 런타임 내부 상태는 세션 간 자동으로 이어지지 않는다.
- 전략 판단 단위는 and 조건의 체인 기반으로 구성할 수 있어야 한다.
- 주문 수량과 전략 내부 판단은 동일 개념이 아니며, 분리된 책임으로 다뤄야 한다.
- 시장 세션 전환(장전, 장중, 장마감)은 시스템 전반의 중요한 상태 변화로 취급한다.

이 제약은 구현체가 바뀌더라도 유지되어야 하는 운영 모델에 가깝다.

## 4) 큰 그림(한눈에 보기)

```text
[User / Order UI]
       |
       v
[Order Config Service] ---> (validated trading config)
       |
       v
[Trading Orchestrator] ---> [Strategy Runtime] ---> (execution request)
       |                           |
       |                           v
       |                    [Runtime State Store]
       v
[Execution Service] <----> [Broker Gateway] <----> [Broker API]
       |
       v
 (execution result / balance / fills)
```

핵심 포인트:

- `Orchestrator`는 순서를 관리한다.
- `Strategy`는 판단한다.
- `Execution`은 실행한다.
- `Gateway`는 브로커 차이를 흡수한다.
- `Config/State`는 입력과 상태를 관리한다.

## 5) 컴포넌트별 R&R

### 5.1 Trading Orchestrator

- 루프 주기와 실행 순서를 통제한다.
- 장전/장중/장마감 상태 전환을 관리한다.
- 어떤 시점에 무엇을 읽고, 무엇을 멈추고, 무엇을 저장할지 결정한다.
- 장애 처리 정책에 따라 재시도, 중단, 다음 주기 이월 중 하나를 선택한다.

### 5.2 Order Config Service

- 사용자 설정(종목, 수량 단계, tick 등)을 읽는다.
- 잘못된 입력을 검증하고 실행 가능한 설정 형태로 정리한다.
- 동기화 시점과 반영 규칙을 책임진다.
- 입력 매체가 바뀌어도 상위 컴포넌트에는 동일한 의미의 설정을 제공해야 한다.

### 5.3 Strategy Runtime

- 현재 상태 + 설정으로 매수/매도 결정을 내린다.
- 전략 내부 상태(스테이지, 체인 상태, 조건 평가 결과)를 유지한다.
- 체인 기반 전략 구성을 수용할 수 있어야 한다.
- 판단 결과를 실행 계층이 소비할 수 있는 형태로 넘긴다.

### 5.4 Execution Service

- 전략 판단 결과를 실제 주문 요청으로 변환한다.
- 주문 정책(재시도, 현금 부족 대응, 대체 자산 처리, 주문 순서)을 수행한다.
- 성공/실패/부분체결/지연 상태를 상위 계층이 이해할 수 있는 결과로 반환한다.

### 5.5 Broker Gateway

- 브로커 API 상세를 캡슐화한다.
- 시세/잔고/주문/체결 조회의 공통 인터페이스를 제공한다.
- 브로커별 인증, 요청 제한, 응답 차이, 시장별 규칙 차이를 내부에서 처리한다.
- 외부 브로커의 서버 이상, 호출 제한 추정치, 세션 불안정성 같은 제약이 상위 계층으로 직접 새지 않도록 완충 지점 역할을 한다.

### 5.6 Runtime State Store

- 가격 히스토리, 포지션, 주문 상태, 전략 런타임 상태를 보관한다.
- 전략/실행 컴포넌트에 조회·갱신 인터페이스를 제공한다.
- 장마감 저장, 사후 분석, 평가를 위한 최소 상태를 남긴다.

## 6) 대표 시나리오

### 6.1 일반 루프

1. 시장 상태를 확인한다.
2. 필요한 시세/잔고/주문 상태를 갱신한다.
3. 전략이 매수/매도 조건을 평가한다.
4. 주문이 필요하면 실행 계층이 브로커 요청으로 변환한다.
5. 실행 결과를 상태 저장소에 반영한다.
6. 다음 주기까지 대기한다.

### 6.2 운영 중 `order_ui` 수정

1. 사용자가 주문 파라미터를 수정한다.
2. 시스템은 즉시 반영하지 않고 정해진 동기화 시점을 기다린다.
3. 동기화 시점에 새 설정을 검증한다.
4. 유효한 설정만 전략 런타임에 반영한다.
5. 이후 판단부터 새 설정이 적용된다.

원칙:

- 즉시 반영보다 “안전한 동기화 시점 반영”을 우선한다.
- 부분 반영보다 “일관된 단위 반영”을 우선한다.
- 반영 실패 시 불완전한 설정으로 운영을 계속하지 않는다.

### 6.3 장전 처리

1. 장 시작 전 필요한 가격/상태를 준비한다.
2. 주문 가능 시점 전에는 실행보다 상태 초기화를 우선한다.
3. 장 시작 시점부터 전략 판단과 주문 실행으로 전환한다.

### 6.4 장마감 처리

1. 신규 주문을 중단한다.
2. 미체결/잔고/체결 상태를 최종 동기화한다.
3. 평가 또는 후속 계산에 필요한 상태를 정리한다.
4. 결과를 저장하고 루프를 종료한다.

## 7) 왜 이렇게 나눴는가

- 브로커 변경은 `Gateway` 중심 변경이어야 한다.
- 전략 변경은 `Strategy Runtime` 중심 변경이어야 한다.
- 입력 채널 변경은 `Config Service` 중심 변경이어야 한다.
- 운영 주기나 배포 구조 변경은 `Orchestrator` 경계 안에서 흡수되어야 한다.
- 상태 저장 방식 변경은 `State Store` 경계 안에서 흡수되어야 한다.

이 분리는 “하나를 바꿀 때 나머지 영향 최소화”를 목표로 한다.
즉, 이 문서의 목적은 현재 구현을 고정하는 것이 아니라 변경에 강한 경계를 정의하는 데 있다.

## 8) 확장성 가이드

### 8.1 브로커 추가

- 신규 브로커 구현은 `Broker Gateway` 계약을 만족해야 한다.
- 상위 컴포넌트는 브로커 고유 요청 포맷을 직접 알지 않는다.

### 8.2 전략 추가

- 전략은 플러그인처럼 등록 가능해야 한다.
- 전략은 체인 조합을 통해 확장 가능해야 한다.
- 전략 변경이 실행/브로커 로직 변경으로 번지지 않게 한다.

### 8.3 설정 채널 교체

- 입력 매체는 바뀌어도 상위 계층이 기대하는 설정 의미는 유지한다.
- 파일 기반, DB 기반, UI 기반 중 무엇이든 운영 안정성을 해치지 않는 반영 규칙을 가져야 한다.

## 9) 공통 품질 기준

- 재시도 정책은 컴포넌트별 최대 횟수와 타임아웃을 명시한다.
- 주문 중복 방지 정책은 명시적으로 정의한다.
- 설정 반영과 주문 실행 로그는 추적 가능해야 한다.
- 루프 지연(주기 초과)을 감지하고 경고할 수 있어야 한다.
- 잘못된 운영 입력이 전략/브로커 계층으로 전파되지 않도록 검증 경계를 둔다.
- 장중 상태와 장마감 상태의 처리 규칙은 혼합되지 않아야 한다.
